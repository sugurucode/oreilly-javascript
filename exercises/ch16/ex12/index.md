# C10K問題とNode.jsの解決策

## 1. C10K問題とは

- **定義**: クライアント（**C**lient）が**1万**（**10K**）台を超えると、ハードウェア性能に余裕があってもサーバのレスポンスが極端に低下する問題。
- **原因**: 従来の「1接続 = 1プロセス/スレッド」モデルでは、OSのメモリ消費と**コンテキストスイッチ**（処理切り替え）の負荷が限界に達するため。

## 2. Node.jsの戦略：シングルスレッド＋非同期I/O

- **アプローチ**: 1つのスレッドで全ての接続をイベントループにより管理。
- **メリット**: プロセス生成のオーバーヘッドがなく、低メモリで大量の同時接続を維持できる。

## 3. 子プロセス（child_process）の役割

シングルスレッドの弱点（**CPU集約型処理でのブロッキング**）を補うために、子プロセスを活用する。

| 手法                 | 特徴                            | 活用シーン                       |
| -------------------- | ------------------------------- | -------------------------------- |
| **`exec` / `spawn**` | 外部OSコマンドを実行            | シェル機能や外部ツールの利用     |
| **`fork`**           | 新しいNode.jsプロセスを起動     | 重い計算処理のバックグラウンド化 |
| **`cluster`**        | `fork`の応用。1つのポートを共有 | マルチコアCPUの全リソース活用    |

## 4. 現代的なシステム構成

現在は、アプリ内の `fork` だけでなく、インフラ全体で負荷を分散するのが一般的。

1. **Nginx（リバースプロキシ）**: 前段で大量の接続（C10K）を受け流す。
2. **Node.js**: シングルスレッドの特性を活かし、非同期にI/Oを処理。
3. **複数インスタンス**: コンテナ（Docker等）を用いて、CPUコア数に合わせてスケールアウトさせる。
